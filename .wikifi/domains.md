# Domains and Subdomains

The system is organized around four primary bounded contexts that govern the automated generation of technical documentation from source repositories. These domains separate structural analysis, semantic extraction, output synthesis, and external intelligence integration, ensuring clear architectural boundaries regardless of underlying implementation details.

### Core Domains and Subdomains

| Domain | Subdomain | Primary Responsibility |
|---|---|---|
| **Repository Introspection & Analysis** | Artifact Filtering & Navigation | Identifies relevant source artifacts while excluding scaffolding, tests, and third-party dependencies. |
| | Structural Analysis & Scope Definition | Maps repository layout and establishes analysis boundaries. |
| | Purpose Inference | Derives a concise functional description from structural patterns and manifest metadata. |
| **Content Extraction & Translation** | Semantic Mapping | Translates per-file structures into domain-relevant insights and documentation-ready mappings. |
| | External Model Analysis | Leverages external analytical capabilities to interpret complex patterns and generate structured observations. |
| **Documentation Synthesis & Lifecycle** | Section Aggregation | Consolidates extracted insights into coherent, structured documentation sections. |
| | Workspace Orchestration | Manages environment setup, state tracking, and pipeline lifecycle handling. |
| **External Intelligence Abstraction** | Provider Decoupling | Isolates core documentation logic from specific external model implementations. |
| | Interaction & Output Generation | Handles communication protocols and formats both structured and unstructured outputs. |

### Inter-Domain Relationships

- **Ingestion to Analysis**: The *Artifact Filtering* subdomain serves as the primary data ingestion layer, supplying filtered paths and structural metadata directly to the *Repository Introspection* domain for scope definition and purpose inference.
- **Extraction to Synthesis**: Insights and mappings generated by the *Content Extraction* domain are routed to *Documentation Synthesis*, where they are consolidated into final documentation sections.
- **Supporting Intelligence Layer**: The *External Intelligence Abstraction* domain operates as a generalized supporting context. It provides analytical capabilities to both extraction and synthesis workflows while maintaining strict decoupling from core business logic.
- **Lifecycle Envelope**: *Workspace Orchestration* wraps the entire pipeline, managing environment state and lifecycle transitions across all active domains.

### Observations and Gaps

- The source notes indicate overlapping responsibilities between the extraction and aggregation workflows, particularly regarding how external model analysis feeds into section synthesis. The exact boundary between semantic mapping and external model analysis is not explicitly defined.
- While the architecture implies a clear separation of concerns, the notes do not formally classify domains as core, supporting, or generic. All four areas appear to be treated as equally critical to the documentation pipeline.
- No direct contradictions were identified, but further clarification is needed on error handling and fallback mechanisms when external analytical capabilities are unavailable or rate-limited.
